現在ご検討いただいておりますAWS S3への移行に関しまして、ファイル連携の構成についてご相談させてください。

現状のファイル連携フローは以下の通りです。
① 外部システム → NASサーバーへファイル配置
② 本システムバッチ → NASからFTPサーバーへファイル転送
③ フロントエンドサービス → FTPサーバーからファイル取得

このフローをAWS S3へ移行するにあたり、以下の2案を検討しております。

【案A】S3バケットを2つに分ける構成
　S3-A（外部システム用） → バッチ処理 → S3-B（フロントエンド用）
　メリット：システム間の依存が切り離され、疎結合となる。障害の影響範囲を限定しやすい。
　デメリット：バケット間のデータ転送処理が発生し、構成・運用がやや複雑になる。

【案B】S3バケットを1つに統合する構成
　外部システム、バッチ、フロントエンドが同一S3バケットを共用
　メリット：システム構成がシンプルで、データ転送処理が不要。コスト低減が期待できる。
　デメリット：システム間の結合度が高く、設定変更・障害時の影響が広がる可能性がある。

つきましては、以下の点についてお客様のご意向を伺えますと幸いです。

１．今回のシステムにおいて、システム間の独立性（疎結合）はどの程度重視されますか？
　（例：外部システムの変更がフロントエンドに影響しないことを重視するか）

２．コスト面（S3バケット数、データ転送量、運用管理工数）の優先度はいかがですか？
　（案Bのほうが月額コスト、運用負荷ともに低くなる想定です）

３．将来的に外部システムやフロントエンドの構成変更・リプレイスの可能性はありますか？
　（変更が見込まれる場合は案Aのほうが柔軟に対応しやすいと考えます）

４．セキュリティやアクセス制御の要件について、システムごとに明確な分離が必要ですか？
　（バケットを分けたほうがIAMポリシー等の管理はシンプルになる面もあります）

５．その他、ファイル連携の方式についてご要望や制約事項はございますか？
